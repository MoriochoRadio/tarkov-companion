# Tarkov Companion — 설계서

> 작성: 2026-06-11 · 상태: 초안 v1

## 1. 한 줄 정의

Escape From Tarkov 플레이어를 위한 **AI 큐레이션 컴패니언 웹** — 실시간 플리마켓 시세·가성비 분석 대시보드 + 매일 자동 생성되는 한국어 브리핑(패치/커뮤니티 꿀팁/메타 변화).

## 2. 차별화 (기존 tarkov.dev, tarkov-market과 다른 점)

1. **AI 일일 브리핑** — 패치노트 + 커뮤니티 정보를 매일 AI가 요약 배달 (기존 사이트 없음)
2. **해석하는 시세** — 단순 나열이 아닌 "오늘 슬롯당 가치 톱", "급등/급락 경보" 등 분석 제공
3. **한국어 우선** — 기존 도구는 전부 영어 중심

## 3. 아키텍처

```
[방문자 브라우저] ──직접 호출──> api.tarkov.dev/graphql (무료, 키 불필요)
       │
       └─ GitHub Pages (정적 호스팅, 무료)
              ▲
              │ 커밋 → 배포 dispatch (매일 09:00 KST)
[GitHub Actions daily-briefing] ── 수집 → GitHub Models 요약 → public/data/briefings/YYYY-MM-DD.json
```

- 서버 없음. 시세는 방문자 브라우저가 API를 직접 호출 → 운영비 0원
- 브리핑은 GitHub Actions가 매일 생성해 리포에 커밋 → 배포 워크플로우를 dispatch해 사이트에 게시 (GitHub 안에서 완결, 구독·유료 의존 없음)
- 브리핑 소스: EFT 위키 체인지로그(MediaWiki API, 공식 패치노트 수록) + Reddit r/EscapefromTarkov 일간 인기글(RSS). 공식 뉴스 페이지는 JS 렌더링 SPA + 내부 API 403이라 직접 수집 불가(2026-06-11 확인) → 위키 체인지로그로 대체. 디시/인벤은 약관 이슈로 보류

## 4. 기술 스택 (전부 무료)

| 영역 | 선택 | 이유 |
|---|---|---|
| 프론트엔드 | React + TypeScript + Vite | 표준 스택, 유지보수 용이 |
| 데이터 | tarkov.dev GraphQL API | 무료·키 불필요·한국어 지원(lang: ko) |
| 호스팅 | GitHub Pages | 무료, 리포 push만으로 배포 |
| CI/CD | GitHub Actions | push 시 자동 빌드·배포 (무료) |
| 브리핑 생성 | GitHub Actions + GitHub Models | GITHUB_TOKEN만으로 무료 AI 요약, 리포 안에서 완결 |

- [x] **버그(최우선) — 중복 퀘스트**: 래그맨 "New Beginning"이 좌측에 3중 표시 + 우측 FIR **3배 과다 집계**. 원인은 **API가 같은 (영어명+상인) 퀘스트를 여러 id로 반환**(게임모드·이벤트 변형) — 실측 510개 중 8그룹 11건 중복("New Beginning" 4 id, "Make Amends" 3 id, "Drip-Out/Textile" 등). id 기준 dedup은 불가(전부 다른 id)라 **`mergeTasks`에서 `(영어명+상인)` 기준 전역 dedup** 신설(`dedupeTasks`): 대표는 **목표 수가 가장 많은 것**(New Beginning은 14목표=FIR 피규어 10종 보유 변형이 선정됨을 실측 확인), `requires`는 대표 id로 재매핑하고 `unlocks`는 대표 집합에서 재계산. 전역 수정이라 퀘스트 탭·준비물·플래너의 중복/과다집계도 동시 해결
- [x] **하이드아웃 레벨별 상세**: 하이드아웃 탭 스테이션 행을 펼치면(`▸/▾`) 레벨별 요구 아이템을 "N레벨 — 아이콘+한/영+×수량" 으로 표시(`levels[].itemRequirements`, 화폐 제외). **이미 지은 레벨은 어둡게+빗금**(기존 `tc:hideout-built` 연동), FIR 요구는 배지
- [x] **퀘스트 아이템 체크 연동**: 좌측 퀘스트 행의 필요 FIR 아이템마다 `− 보유 +` 스테퍼(P27 우측 그리드와 같은 방식). `tc:prep-counts` 공유라 **좌측에서 바꾸면 우측 정크박스 그리드도 실시간 반영**(반대도). 필요수 충족 항목은 취소선·흐리게
- [x] **성능(CLAUDE.md)**: 스테퍼가 붙어 무거워진 좌측 아코디언에 **상인 전환 2단계 렌더**(`QUEST_FIRST_PAINT=8`) 추가. CPU 4x 스로틀 실측 — 무거운 상인(래그맨) 전환 90ms/rAF 2ms, 스테이션 펼침·스테퍼 등 FIR 조작 전부 rAF ≤7ms(<1s). 모바일은 아이템 행이 한 줄씩(이름 말줄임)이라 스테퍼 정렬 유지

### Phase 32 — 맵 뷰어 화질: 줌 정지 시 벡터 재래스터 (완료)

문제: 맵을 확대하면 뭉개짐. 소스 탓이 아님 — `public/maps/*.svg`는 `<image>` 0개 진짜 벡터. 진짜 원인은 `.mapview-layer`가 `will-change:transform` + `scale()`로 **GPU 합성 레이어**가 돼 SVG를 기본 크기(rect ~300px) 비트맵 텍스처로 한 번만 굽고, 확대 시 그 텍스처를 컴포지터가 늘리기만 함 → 벡터인데도 14배에서 저해상 스냅샷이 늘어나 픽셀 뭉개짐.

- [x] **배율 분리(`view.s` vs `baseScale`)**: `baseScale` = SVG가 실제로 래스터된 배율. 제스처 중 transform = `scale(s/baseScale)`로 차이만 compositor가 확대(부드럽지만 흐림). 레이어 CSS 크기를 `rect*baseScale`로 두면 SVG(host 100%)가 그 해상도로 래스터됨
- [x] **줌 정지 commit (디바운스 150ms)**: 멈추면 `baseScale=min(s, cap)`으로 올려 setState → 레이어 크기·마커 위치 재계산, `useLayoutEffect`에서 transform을 `s/baseScale`로 동기(같은 페인트 → 튐 없음). 팬(translate)만 할 땐 commit 트리거 안 함
- [x] **마커 정합**: 마커 left/top을 `(px-minX)*baseScale`, 역스케일 `--inv=baseScale/s`로 — SVG와 같은 baseScale 공간이라 재래스터 전후 위치 동일(실측: commit 전후 svg 화면폭 3580px·svgL −1269 불변)
- [x] **메모리 상한(폴백)**: `baseScaleCap = sqrt(24M / (rect.w*rect.h*dpr²))` (~96MB). rect이 작아(최대 345px) 대부분 맵은 최대 줌까지 캡에 안 걸려 완전 또렷. 큰 맵·고배율만 캡에서 compositor scale로 폴백(streets/shoreline은 baseScale ~13에서 캡)
- [x] **성능(CLAUDE.md)**: 재래스터는 Chrome 래스터 스레드에서 off-main → commit의 메인스레드 비용은 setState+마커 리렌더+레이아웃뿐. CPU 4x 실측 — 세관 commit rAF 45ms, **해안선(304K) 고배율 commit rAF 4ms**, 단일 롱태스크 <1s. 1초 블로킹 없음
- 메모: 헤드리스 스크린샷은 GPU 합성 transform을 못 잡아 풀맵처럼 찍힘 — 화질 검증은 DOM 기하(표시 크기 대비 텍스처 해상도 0.28×→0.96×)로 확인. 기존 localStorage 키 불변(건드린 것 없음)

### Phase 34 — 플래너 맵 마커: 정밀 위치 + 마커 정리 (완료)

친구 피드백: 마커가 "맵 전체에 점으로만" 떠 정확한 위치를 모르겠고, 퀘스트 5개+ 켜면 헷갈림. 친구는 이걸 "Phase 33"으로 지칭했으나 33은 모딩 콜아웃이 씀 → 34로 기록. **위키 위치 사진 재호스팅은 저작권상 금지(원칙 유지)** → 우리 자체 맵(라이선스 보유)에서 확대·강조로 해결, 정확한 디테일은 출처 **링크**로만. 데이터 한계: 목표는 좌표 "점"(loc.x,z)만 있고 구역 폴리곤 없음 → "구역 강조"는 그 점 자동 확대 + 강조 링으로 근사.

- [x] **① 클릭 자동확대(`focusMarker`/`animateTo`)**: 마커 클릭 시 그 좌표를 화면 중앙으로 `fit*7`(ZOOM_MAX 내) 부드럽게 이동. **rAF로 transform/`--inv`만 갱신(레이아웃 패스 0)** → 저사양에서도 가벼움. 정지 후 `commit()`으로 그 배율 벡터 재래스터(Phase 32 흐름 유지). 손대면(`pointerdown`/`zoomAt`) 즉시 `cancelAnim`. reduced-motion이면 애니 없이 즉시 이동
- [x] **① 강조 링(`.mapmark-focus`)**: 포커스 마커 위에 상시 링(`::before`) + 펄스 2회(`::after`, `mapmark-pulse`). reduced-motion이면 펄스 없이 정적 링만. 마커와 같은 baseScale 공간 + `scale(--inv)`로 화면 크기 고정
- [x] **① 출처 링크(팝오버)**: "위키에서 정확한 위치 ↗"(`quest.wikiLink`) + "tarkov.dev 맵 ↗"(`mapNormalizedName`) + "근사 위치 — 정확한 지점은 출처 링크 참고" 문구. PlannerTab이 `ViewMarker`에 `wikiLink`·`mapNormalizedName`를 실어 보냄(MapViewer는 quests API 무지 유지). **팝오버는 맵 상단에 앵커** — 자동확대로 마커가 중앙에 오므로 중앙의 마커·링을 가리지 않게(정확한 위치 확인이 목적)
- [x] **②A 퀘스트별 마커 표시 토글**: 좌측 리스트 색 점 → "표시 토글" 버튼(`.mapmark-eye`, off면 점 흐리게+빗금). 끄면 그 퀘스트 마커만 맵에서 숨김(픽·가방·체크 그대로). `label` 안의 버튼이라 `preventDefault`로 체크박스 토글 방지. 맵 위 "표시 중 N / 전체 M" + "모두 표시" 복구 버튼

- [x] **단일 제출 아이템만 카운터(`submitItem`)**: 통합 체크리스트 `buildRows`와 동일 기준(giveItem/plantItem · `items.length===1` · 비화폐)으로 추적 대상 판별 → 같은 itemId로 `usePrepCounts`에 기록돼 일관. "여러 아이템 중 하나" 선택형은 특정 아이템 지목 불가라 카운터 없음(기존 칩 표시 유지)
- [x] **UI**: 아이템마다 `[− 보유 N +]` + "· 요구 ×{count}"(이 퀘스트 요구량), 보유 ≥ 요구면 초록 ✓. 버튼은 기존 `.prep-step` 재사용. 섹션 상단에 "통합 체크리스트와 같은 저장소" 안내. 보유는 전역(공유) 값이라 분수 대신 "보유/요구"를 분리 표기해 혼동 방지
- [x] **검증**: "신상이 왔어요" 상세에서 + 클릭 → 보유 0→증가, `tc:prep-counts` 기록(통합 체크리스트와 동일 스토어라 자동 동기화), 보유≥요구 시 met ✓ 노출 실측. 빌드 통과, `profile-ui` 상세 진입 회귀 없음(rAF 11ms·최대 롱태스크 470ms<1초)
- 메모: 이로써 "어디서든 퀘스트 완료 체크(Phase 39) + 어디서든 필수 아이템 체크(Phase 40)"가 한 `prep-counts`·`done-quests`로 묶임. 남은 후속: 제작/바터→돈벌이 딥링크 · 다음 퀘스트 로드맵 · 중복 5뷰 로직 통폐합

### Phase 41 — 필요템 → 돈벌이(제작·바터) 딥링크 (완료)

Phase 36에서 미뤘던 v1.1. 통합 체크리스트에서 *사야 할* 아이템을 *만들/바꿀* 길로 연결 → "돈벌이" 탭에 열 이유를 부여(척추가 탭을 하나 더 끌어들임). 시세(검색)·퀘스트(상세) 딥링크에 이어 세 번째.

- [x] **출력 인덱스(`craftBarterOutputIds`)**: 돈벌이 데이터(crafts·barters)의 산출 아이템 id 집합 → 그 아이템에만 "🔁 제작·바터" 링크 노출(없는 아이템엔 안 띄움). 통합 체크리스트가 **별도 비차단 로드**(`useAsyncData(fetchProfitData)`)로 받아 첫 페인트는 안 막고 링크만 나중에 채움
- [x] **pending 네비(`setPendingProfit`/`consumePendingProfit`)**: `setPendingSearch`/`setPendingQuest`와 같은 모듈 변수 패턴. App이 `onProfit`(itemId)를 모든 탭에 전달(`pickProfit`→`switchTab('profit')`); ProfitTab이 마운트 시 consume. PrepChecklist→FirTab 경로로 `onProfit` 전달
- [x] **ProfitTab "이 아이템 만들기/바꾸기" 필터 뷰**: `outputItem` 설정 시 정상 뷰 대신 그 아이템을 산출하는 **크래프트 + 바터를 한 화면에**(모드 토글 우회). "← 전체 돈벌이 보기"로 복귀. 레시피가 목적이라 시세 미확인분도 표시하되 수익만 "시세 일부 미확인"으로 생략(`RecipeRow`에 `incomplete` 플래그)
- [x] **검증**: 데스크톱/375px(`shoot-profit-link.mjs`) — 통합 체크리스트 "5.56 MK 318 SOST"의 🔁 제작·바터 → 돈벌이 탭 점프, 크래프트 1+바터 1 표시, 시세 미확인 처리, 전체 보기 복귀 확인. 빌드 통과, `profile-ui` 회귀 없음(내 진행 진입 rAF 134ms·최대 롱태스크 693ms<1초)
